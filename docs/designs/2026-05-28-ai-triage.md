# AI Triage design â€” single-shot suggestion of priority, labels, acceptance criteria

Date: 2026-05-28
Scope: Backend vertical slice `Features/Ai/TriageIssue`; pluggable `IChatClient` wiring; BFF passthrough; an "AI-suggest" action inside the issue edit form that fills the form with suggestions. Plus removal of the now-dead `AcceptanceCriteriaAiSuggested` flag.
Excludes: AI PR Reviewer (separate design).
Depends on: 2026-05-24-projects, 2026-05-26-issues.

## Decisions

- **Trigger:** an "AI-suggest" button **inside `<IssueDetail>`'s Edit mode form** â€” not on the board card. The open form is the surface; nothing new opens. The read-mode "AI-suggested" badge sketched in the Issues design is dropped.
- **Flow:** quick-create (title only) â†’ open issue â†’ Edit mode â†’ click "AI-suggest" â†’ the form fields fill with suggestions â†’ user tweaks â†’ Save (the normal PUT). Nothing persists until Save.
- **No DB writes** in the triage slice, and **no `ai_suggested` flag** â€” it's redundant because the user always reviews and saves via the existing update path. The `AcceptanceCriteriaAiSuggested` column/field is removed.
- **Input = live form values.** The endpoint takes a `{ title, description }` body so the user can type a description and triage on it before saving. Title/description analyzed come from the body, not the DB.
- **Labels resolved server-side.** The model suggests label names; the handler matches them (case-insensitive) against the project's real labels and returns full `LabelSummary` objects. Hallucinated labels are silently dropped. (CLAUDE.md sketched `string[]`; resolved objects are returned instead so the form can map directly to label ids.)
- **Apply semantics in the form:** priority â†’ set; labels â†’ **union** into current selection (dedupe by id, never removes user picks); acceptance criteria â†’ **overwrite** the textarea.
- **Pluggable LLM.** A single `OpenAI`-provider registration drives both local Ollama (OpenAI-compatible at `/v1`) and any cloud OpenAI-compatible endpoint; switch by config only.
- **Defensive parsing.** Priority is clamped to the enum (fallback = issue's current priority); labels filtered to real ones; criteria trimmed and length-capped. A sloppy model can't produce an invalid suggestion.

## Endpoint

`RequireUser` policy. Project resolved by `slug` first (404 `projects:project:not_found`).

| Method | Route | Purpose |
|--------|-------|---------|
| POST | `/api/projects/{slug}/issues/{number}/ai/triage` | Single-shot triage of the supplied title/description |

> Route follows the shipped project-scoped-by-number convention. CLAUDE.md's `/api/issues/{id}/ai/triage` is stale.

**Request body:**

```json
{ "title": "string (1â€“200)", "description": "string (â‰¤10000, optional)" }
```

**Response (`200`):**

```csharp
public record TriageSuggestion(
    string Priority,            // kebab enum: low | medium | high | urgent
    LabelSummary[] Labels,      // resolved against the project (id, name, color)
    string AcceptanceCriteria); // markdown
```

`LabelSummary` is the existing `{ id, name, color }` DTO reused from the Issues slice.

**Handler flow** (`Features/Ai/TriageIssue.cs`, MediatR as usual):

1. Resolve project by `slug` â†’ 404 `projects:project:not_found`.
2. Verify issue exists by `number` â†’ 404 `issues:issue:not_found`. (Existence check only â€” the analyzed content comes from the body, not the row.)
3. Load the project's labels.
4. Build the prompt (below), call `IChatClient`.
5. Parse JSON defensively â†’ resolve priority + labels, trim criteria.
6. Return `TriageSuggestion`. **No persistence.**

## Prompt & LLM call

`Microsoft.Extensions.AI` `IChatClient`.

- **System prompt:** fixes the role ("you triage software issues"), enumerates the four priorities, and hard-constrains label choice: *choose labels only from the provided list; if none fit, return an empty array; never invent labels*. Specifies strict-JSON output matching the schema and nothing else.
- **User message:** project name + description, the project's label set (names), and the issue title + description **from the body**.
- `ChatResponseFormat.Json` when the provider supports it; low temperature (~0.2) for stable demo output; a per-request timeout guards the single call.

**Defensive resolution (don't trust the model):**

- Deserialize into an internal `{ priority, labels[], acceptanceCriteria }`. On parse failure, retry once; still failing â†’ 502 `ai:triage:llm:invalid_response`.
- **Priority** â†’ map to enum; off-list value falls back to the issue's current priority.
- **Labels** â†’ case-insensitive match against the project's real labels; keep hits as `LabelSummary`, drop hallucinations.
- **Acceptance criteria** â†’ trim, defensively cap at 10 000 chars.

Worst case for a sloppy local model is empty labels / fallback priority â€” the frontend always receives clean, applicable values.

## Provider wiring (pluggable `IChatClient`)

Ollama exposes an OpenAI-compatible API at `/v1`, so one registration through the `OpenAI` provider serves both local and cloud â€” only config changes.

```csharp
services.AddChatClient(sp =>
    new OpenAIClient(
            new ApiKeyCredential(opts.ApiKey),               // "ollama" placeholder locally
            new OpenAIClientOptions { Endpoint = new(opts.BaseUrl) })
        .GetChatClient(opts.Model)
        .AsIChatClient())
    .UseLogging();
```

`LlmOptions { BaseUrl, Model, ApiKey }` bound from configuration.

**Aspire:** AppHost adds an `llm` connection-string parameter (base URL) plus `Llm__Model` / `Llm__ApiKey` via config/user-secrets. Switch localâ†”cloud by changing the parameter â€” no code change. Optionally AppHost can spin an Ollama container in dev; not required for the slice.

## Errors

All RFC 7807 ProblemDetails with `errorCode` + `traceId`, passed through the BFF unchanged.

| Code | When | HTTP |
|------|------|------|
| `projects:project:not_found` | unknown slug | 404 |
| `issues:issue:not_found` | unknown number | 404 |
| `ai:triage:title:required_or_too_long` | body title empty / >200 (FluentValidation) | 400 |
| `ai:triage:description:too_long` | body description >10 000 | 400 |
| `ai:triage:llm:unavailable` | LLM call throws (network / timeout / auth) | 502 |
| `ai:triage:llm:invalid_response` | JSON unparseable after one retry | 502 |

## Frontend

**BFF passthrough route:** `app/api/projects/[slug]/issues/[number]/ai/triage/route.ts` â€” `POST` handler; attaches `X-Bff-Secret` + `X-User-Id`, forwards the `{ title, description }` body, returns the response (including ProblemDetails) unchanged. No reshaping.

**"AI-suggest" button** inside `<IssueDetail>`'s Edit mode form (secondary/ghost button with a sparkle glyph, styled per the `ui-design` skill). Not on the board card.

**Click behavior:**

1. Disabled when the title field is empty (mirrors the backend title-required rule).
2. On click â†’ loading state (spinner, disabled); POST the **live** form `{ title, description }` to the BFF route.
3. On `200`, merge into the open form state (no save):
   - **priority** â†’ set the priority dropdown.
   - **labels** â†’ **union** suggested `LabelSummary[]` into current selection, dedupe by id (keeps user picks).
   - **acceptanceCriteria** â†’ **overwrite** the criteria textarea.
4. Form stays in Edit mode; user tweaks â†’ **Save** (existing PUT).
5. On error â†’ toast via the shared `errorCode â†’ message` map; form untouched; button returns to idle.

Re-clicking re-runs triage (no writes, idempotent).
