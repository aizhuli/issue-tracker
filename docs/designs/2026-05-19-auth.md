# Auth design â€” BFF-only session cookies

Date: 2026-05-19
Scope: Backend (ASP.NET) + Next.js BFF route handlers. No UI pages.

## Decision

Drop JWT and refresh tokens. The API is only reachable by the Next.js BFF
(Aspire wires the URL via service discovery), so the API does not need a
portable token format. Instead:

- **Browser â†’ BFF:** encrypted `iron-session` cookie (HTTP-only, SameSite=lax).
  Session shape: `{ userId, email, name }`.
- **BFF â†’ API:** every request carries `X-Bff-Secret: <env>`. Authenticated
  requests also carry `X-User-Id: <base32 id>`.
- **API:** custom `AuthenticationHandler` validates the secret and maps
  `X-User-Id` into a `NameIdentifier` claim. Single auth scheme.

Logout is purely a BFF concern â€” there is no server-side session state to
destroy.

This deviates from `CLAUDE.md`'s JWT + refresh-token design. The rationale
is MVP simplicity for a single trusted BFF.

## Backend

### Foundation (built once, shared across all future slices)

- `Common/Identity/IdFactory.cs`, `Base32Encoder.cs`, `IdEncoding`.
- `Common/Http/IEndpoint.cs` + `MapEndpoints()` assembly-scan extension.
- `Common/Validation/ValidationBehavior.cs` â€” MediatR pipeline; throws
  `ValidationException` which the global handler converts to 400 with
  ProblemDetails.
- `Common/Exceptions/` â€” `DomainException`, `ConflictException`,
  `UnauthorizedDomainException`, `NotFoundException`,
  `GlobalExceptionHandler`.
- `Common/Auth/` â€” `BffAuthOptions`, `BffAuthenticationHandler`,
  `ICurrentUser`/`CurrentUser`, `PasswordHasher`.
- `Data/AppDbContext.cs` + `Data/Entities/User.cs` + initial migration.

### Endpoints

| File                          | Route                  | Auth                  |
|-------------------------------|------------------------|-----------------------|
| `Features/Auth/Register.cs`   | `POST /api/auth/register` | BFF secret           |
| `Features/Auth/Login.cs`      | `POST /api/auth/login`    | BFF secret           |
| `Features/Auth/Me.cs`         | `GET /api/auth/me`        | BFF secret + user id |

Request/response DTOs use Base32-encoded IDs. Passwords hashed with
`PasswordHasher<User>` (PBKDF2).

### Error codes

- `auth:user:email:already_exists` (409, Register)
- `auth:credentials:invalid` (401, Login)
- `auth:user:not_found` (404, Me when the BFF sends a stale id)

## BFF (Next.js)

- `lib/session.ts` â€” iron-session config; cookie name `ait_session`.
- `lib/api-client.ts` â€” `serverFetch(path, init)`, attaches
  `X-Bff-Secret` and (when session present) `X-User-Id`.
- `middleware.ts` â€” protects non-`/login`, non-`/register`, non-`/api/auth/*`
  routes. UI pages do not exist yet but the matcher is in place.
- `app/api/auth/register/route.ts` â€” proxies to backend; on 200, saves
  `{ userId, email, name }` into session.
- `app/api/auth/login/route.ts` â€” same against `/login`.
- `app/api/auth/logout/route.ts` â€” `session.destroy()`, returns 204.
- `app/api/auth/me/route.ts` â€” proxies to backend.

Required package: `iron-session`.

ProblemDetails responses pass through unchanged.

## Aspire wiring

- AppHost generates a per-run BFF secret parameter and injects it as
  `BffAuth__SharedSecret` on the API and `BFF_SHARED_SECRET` on the
  frontend.
- AppHost generates a per-run iron-session password and injects it as
  `SESSION_COOKIE_PASSWORD` on the frontend.
- API base URL passed to the frontend via Aspire's service discovery
  (`services__api__http__0`).

## Out of scope for this slice

- Password reset, email verification, OAuth, account lockout, MFA.
- JWT, refresh tokens, server-side session revocation.
- Frontend UI (login/register pages, layouts).
- BFF unit/E2E tests.
