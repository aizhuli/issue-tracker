# Issues design — Kanban dashboard + full CRUD + comments + labels

Date: 2026-05-26
Scope: Backend vertical slices for Issues + Labels + Comments + the board API; BFF passthrough; frontend Kanban dashboard, detail modal, and dedicated issue page.
Excludes: AI Triage, AI PR Reviewer (separate designs).
Depends on: 2026-05-19-auth, 2026-05-24-projects.

## Decisions

- **Primary view:** Kanban board with five columns — `backlog / todo / in-progress / in-review / done`. Done is **collapsed by default** behind a toggle in the viewbar. Each column is independently token-paginated (AIP-158); "Load more" lives at the bottom of each column.
- **Status changes:** drag-and-drop using `dnd-kit`, with a keyboard/menu fallback exposed via a small caret on every card (opens a status dropdown). The board issues `PATCH /api/projects/{slug}/issues/{number}/status` optimistically and rolls back on failure.
- **Detail view:** clicking a card opens a **centered modal** (quick look, no URL change). The modal has an "Open ↗" button that navigates to the **dedicated page** `/projects/{slug}/issues/{number}`. Both shells render the same `<IssueDetail>` component.
- **Issue identifier:** per-project sequential integer `Number`, displayed as `{projectSlug}_{number}` (e.g., `web-platform_42`). URLs use the number. Base32 `id` remains the primary key in DTOs.
- **Create:** single `+ New issue` button in the viewbar, title-only quick-create. Always lands in `backlog`. Editing the rest happens later via the detail view's Edit mode.
- **Edit model:** detail view is **read-only by default**; the "Edit" button switches to a form with Save / Cancel. One PUT writes all editable fields. Status and assignee each have their own PATCH endpoints used by DnD and quick-assign.
- **Board filters:** text search (title + description, case-insensitive substring), multi-select priority, multi-select assignee (includes "Unassigned"). No label filter in MVP.
- **Labels:** managed entirely inline from the issue's label picker — create, rename, recolor, delete. No dedicated settings page. Project owner gate on rename/delete; anyone authenticated can create.
- **Comments:** markdown body, author-only edit/delete. Sorted ascending by `created_at`. Rendered with `react-markdown` + `remark-gfm`. PR links live in comments — there is no dedicated PR URL field on the issue.
- **Delete:** an issue can be deleted by its reporter or the project owner. Cascades to comments and the `IssueLabel` join.
- **Markdown editing:** plain monospace textarea on edit, GitHub-flavored rendering on read. No toolbar in MVP.

## Data model

Three new entities (`Issue`, `Label`, `Comment`) plus the `IssueLabel` join. One new column on `Project` for the issue-number counter.

```csharp
// Data/Entities/Issue.cs
public class Issue
{
    public long Id { get; set; }                      // IdGen
    public long ProjectId { get; set; }
    public Project Project { get; set; } = null!;
    public int Number { get; set; }                   // per-project sequential
    public string Title { get; set; } = "";           // 1–200
    public string? Description { get; set; }          // 0–10_000, markdown
    public IssueStatus Status { get; set; }           // backlog/todo/in-progress/in-review/done
    public IssuePriority Priority { get; set; }       // low/medium/high/urgent
    public long? AssigneeId { get; set; }
    public User? Assignee { get; set; }
    public long ReporterId { get; set; }
    public User Reporter { get; set; } = null!;
    public string? AcceptanceCriteria { get; set; }   // 0–10_000, markdown
    public bool AcceptanceCriteriaAiSuggested { get; set; } // for AI Triage later
    public DateTimeOffset CreatedAt { get; set; }
    public DateTimeOffset UpdatedAt { get; set; }
    public DateTimeOffset? ClosedAt { get; set; }     // set on enter-Done, cleared on exit

    public ICollection<Comment> Comments { get; set; } = [];
    public ICollection<IssueLabel> IssueLabels { get; set; } = [];
}

public enum IssueStatus { Backlog, Todo, InProgress, InReview, Done }
public enum IssuePriority { Low, Medium, High, Urgent }

// Data/Entities/Label.cs
public class Label
{
    public long Id { get; set; }
    public long ProjectId { get; set; }
    public Project Project { get; set; } = null!;
    public string Name { get; set; } = "";            // 1–40, stored lowercase
    public string Color { get; set; } = "";           // hex, e.g. "#6B7280"
    public DateTimeOffset CreatedAt { get; set; }
    public ICollection<IssueLabel> IssueLabels { get; set; } = [];
}

// Pure join — explicit for EF clarity.
public class IssueLabel
{
    public long IssueId { get; set; }
    public Issue Issue { get; set; } = null!;
    public long LabelId { get; set; }
    public Label Label { get; set; } = null!;
}

// Data/Entities/Comment.cs
public class Comment
{
    public long Id { get; set; }
    public long IssueId { get; set; }
    public Issue Issue { get; set; } = null!;
    public long AuthorId { get; set; }
    public User Author { get; set; } = null!;
    public string Body { get; set; } = "";            // 1–10_000, markdown
    public DateTimeOffset CreatedAt { get; set; }
    public DateTimeOffset UpdatedAt { get; set; }
}

// Project gains one column for the per-project counter:
public int NextIssueNumber { get; set; } = 1;
```

### EF configuration highlights

- `Issue`: composite unique index `(ProjectId, Number)`. `HasIndex(ProjectId, Status, CreatedAt DESC, Id DESC)` to power per-column pagination cheaply. `OnDelete(Cascade)` from `Project → Issue`.
- `Issue.AssigneeId` and `Issue.ReporterId`: `OnDelete(Restrict)` — users can't be deleted while owning issues. `Reporter` is required and never changes after create.
- `Label`: unique index on `(ProjectId, Name)`, case-insensitive (`Name` stored lowercased, compared ordinal). Cascade from `Project → Label`.
- `IssueLabel`: composite PK `(IssueId, LabelId)`. Cascade from both sides.
- `Comment`: cascade from `Issue`. `HasIndex(IssueId, CreatedAt, Id)` for stable ordering.
- Enums stored as `string` (`.HasConversion<string>()`) so SQL stays readable and migrations don't shift on enum reordering.

### Number allocation

In `CreateIssue` handler, use a single SQL round-trip:

```sql
UPDATE projects
SET next_issue_number = next_issue_number + 1
WHERE id = @projectId
RETURNING next_issue_number - 1;
```

That atomically reserves the issue's `Number` and avoids a race between concurrent inserts. The unique `(ProjectId, Number)` index is the safety net.

### Closed-at semantics

The `ChangeStatus` handler sets `ClosedAt = utcNow` when transitioning into `Done` from anything else, and clears it when transitioning out of `Done`. The full `UpdateIssue` (PUT) handler applies the same rule when `Status` changes through it.

Migration: `AddIssuesLabelsComments` (single migration covering all three entities, the join table, and `Project.NextIssueNumber`).

## Endpoints

All endpoints sit under the `RequireUser` policy (valid `X-Bff-Secret` + `X-User-Id`). Project-scoped routes resolve the project by `slug` first; unknown slug → `404 projects:project:not_found`.

### Issues

| Method | Route                                                | Purpose                                             |
|--------|------------------------------------------------------|-----------------------------------------------------|
| POST   | `/api/projects/{slug}/issues`                        | Create (title-only quick form or full create)       |
| GET    | `/api/projects/{slug}/issues`                        | List one column (token-paginated, requires `status`)|
| GET    | `/api/projects/{slug}/issues/{number}`               | Get by per-project number                           |
| PUT    | `/api/projects/{slug}/issues/{number}`               | Full edit (Save in Edit mode)                       |
| PATCH  | `/api/projects/{slug}/issues/{number}/status`        | DnD / status menu — body `{ status }`               |
| PATCH  | `/api/projects/{slug}/issues/{number}/assignee`      | Quick assign — body `{ assigneeId \| null }`        |
| DELETE | `/api/projects/{slug}/issues/{number}`               | Delete (reporter or project owner)                  |

The list endpoint is **single-column**: `status` is a required query parameter. The board page issues 4 parallel calls on mount (backlog/todo/in-progress/in-review) sharing the same filter set; the Done toggle adds a 5th call. "Load more" inside a column re-calls the same endpoint with that column's `pageToken`.

Query string: `status` (required, single value), `pageToken`, `maxPageSize` (default 50, cap 100), `q` (text), `priority` (repeatable), `assignee` (repeatable; one value can be the literal `unassigned`).

`IssueSummary` DTO returned by the list:

```json
{ "id": "01J...", "number": 42, "displayKey": "web-platform_42",
  "title": "Login fails on Safari 17", "status": "in-progress",
  "priority": "high",
  "assignee": { "id":"01J...", "name":"Maya", "avatarUrl":null } | null,
  "labels": [{ "id":"01J...", "name":"bug", "color":"#EF4444" }],
  "commentCount": 3, "updatedAt": "..." }
```

`IssueFull` (returned by GET-by-number and create/edit) adds `description`, `acceptanceCriteria`, `acceptanceCriteriaAiSuggested`, `reporter`, `createdAt`, `closedAt`.

**Create** body — minimum is `{ title }`. Optional `description`, `priority`, `assigneeId`, `labelIds[]`, `acceptanceCriteria` round out the full-form variant when AI Triage lands. `status` is **not** accepted in the body; create always lands in `backlog`. Reporter is `ICurrentUser.UserId`.

**Status PATCH:** validates the value against the `IssueStatus` enum, applies the closed-at rule, returns the updated `IssueFull`. No transition restrictions — backward steps and skips allowed per CLAUDE.md.

**Assignee PATCH:** `assigneeId` must reference an existing user; `null` clears the assignment.

### Labels

| Method | Route                                          | Purpose                          |
|--------|------------------------------------------------|----------------------------------|
| GET    | `/api/projects/{slug}/labels`                  | List all labels (no pagination — capped at 200/project) |
| POST   | `/api/projects/{slug}/labels`                  | Create — anyone authenticated    |
| PUT    | `/api/projects/{slug}/labels/{labelId}`        | Rename / recolor — owner only    |
| DELETE | `/api/projects/{slug}/labels/{labelId}`        | Delete — owner only              |

Body shape: `{ name: string (1–40), color: string (hex) }`. Delete cascades to `IssueLabel`. The cap of 200 labels per project keeps the picker UX sane and lets us skip pagination here.

### Comments

| Method | Route                                                              | Purpose                             |
|--------|--------------------------------------------------------------------|-------------------------------------|
| GET    | `/api/projects/{slug}/issues/{number}/comments`                    | List, token-paginated, ASC by time  |
| POST   | `/api/projects/{slug}/issues/{number}/comments`                    | Create — anyone authenticated       |
| PUT    | `/api/projects/{slug}/issues/{number}/comments/{commentId}`        | Edit — author only                  |
| DELETE | `/api/projects/{slug}/issues/{number}/comments/{commentId}`        | Delete — author only                |

`CommentDto`: `{ id, authorId, authorName, authorAvatarUrl, body, createdAt, updatedAt, edited: boolean }`. `edited` is a server-computed convenience (`UpdatedAt > CreatedAt + 1s`).

### Users (assignee picker support)

| Method | Route                                  | Purpose                                  |
|--------|----------------------------------------|------------------------------------------|
| GET    | `/api/users/search?q=...&maxPageSize=` | Prefix match on `name` and `email`, capped at 20 results |

No pagination cursor here — the picker only needs the top 20 matches. The endpoint returns `{ id, name, email, avatarUrl }`.

### BFF passthrough

All of the above mirror under `app/api/projects/[slug]/issues/...` and friends with pure passthrough — attach `X-Bff-Secret` + `X-User-Id`, forward the request, return the response (including ProblemDetails) unchanged. No reshaping, no validation, no caching.

## Frontend

### Routes & BFF

```text
app/(app)/projects/[slug]/page.tsx                 # board (default project landing)
app/(app)/projects/[slug]/issues/[number]/page.tsx # dedicated issue page

app/api/projects/[slug]/issues/route.ts                       # GET, POST
app/api/projects/[slug]/issues/[number]/route.ts              # GET, PUT, DELETE
app/api/projects/[slug]/issues/[number]/status/route.ts       # PATCH
app/api/projects/[slug]/issues/[number]/assignee/route.ts     # PATCH
app/api/projects/[slug]/labels/route.ts                       # GET, POST
app/api/projects/[slug]/labels/[labelId]/route.ts             # PUT, DELETE
app/api/projects/[slug]/issues/[number]/comments/route.ts     # GET, POST
app/api/projects/[slug]/issues/[number]/comments/[commentId]/route.ts # PUT, DELETE
app/api/users/search/route.ts                                 # GET
```

The project card on `/projects` gains an `onClick` that pushes `/projects/{slug}` — the board is the project's home.

### Board page (`/projects/[slug]`)

Standard shell (sidebar + topbar + viewbar). Viewbar contents, left → right:

- Project name as title; small breadcrumb `Projects / web-platform`.
- `.tb-search` input bound to `?q=`, debounced 300ms.
- Priority filter — `.chip-group` with low/medium/high/urgent, multi-select.
- Assignee filter — avatar pill button → opens a popover with searchable user list and an "Unassigned" entry; selected assignees render as a row of stacked avatars on the pill.
- "Show Done" toggle — small ghost button; on click, the Done column slides in.
- `+ New issue` primary button (same style as projects).

Body: a horizontal flex row of `<Column>` components, each 320px wide, gap 16px, the row itself horizontally scrollable. Each column is a vertical flex of:

- Header — name + count + a thin colored top border (one accent per status, sampled from the design tokens).
- Scrollable card stack.
- A "Load more" footer button shown only when `nextPageToken` is non-null.

Column data lives in a single `boardState: Record<IssueStatus, ColumnState>` shape: `{ items, nextPageToken, loading }`. Each column has its own `useEffect` that re-fetches whenever the shared filter set changes; the four columns fetch in parallel via `Promise.all`. Filter changes reset all columns to page 1.

### DnD + status menu

`dnd-kit/core` + `dnd-kit/sortable`. Each card is a `useDraggable`; each column is a `useDroppable` with the column's status id. On drop:

1. Compute `nextStatus` from the drop target.
2. Optimistically remove the card from the source column and prepend to the destination column.
3. `PATCH /status`. On 2xx, refresh both columns' first page in the background to keep counts honest. On failure, roll back the optimistic move and show a toast.

The card also exposes a small caret (`▾`) in its corner that opens a 5-item status dropdown — same network call, no DnD library traversal. This is the keyboard / a11y path.

### `<IssueCard>`

```text
┌─────────────────────────────────┐
│ web-platform_42         ▾       │   ← mono chip · status caret
│ Login fails on Safari 17        │   ← title, 2-line clamp
```

Tokens: `--surface`, `--border`, `--radius`, `--shadow-sm`, hover `--shadow-md`. Priority icon is a small filled glyph with the priority color from the design system. Label chips overflow to "+2" beyond three visible. Click → opens the detail modal.

### Detail shell: modal vs page

`<IssueDetail issue={...} />` is a shared component. Two outer shells:

- **Modal shell** — opened by clicking a card. Centered `.ai-modal`, max-width 720px, max-height 80vh, `ai-pop` entrance, Esc / scrim closes. Header has a small "Open ↗" link that calls `router.push('/projects/{slug}/issues/{number}')` — same component, different shell, real URL.
- **Page shell** — server component fetches `IssueFull` and renders the same `<IssueDetail>` inside the standard app shell. Back button returns to the board.

### `<IssueDetail>`

Read-only by default. Two-column layout at ≥860px, stacked below.

Main column:

- Title (Manrope 600, 20px) + `{slug}_{number}` mono chip beside it.
- Status pill, priority badge, assignee avatar+name as a top metadata row.
- Description — rendered markdown (`react-markdown` + `remark-gfm`). Empty state: muted "No description yet."
- Acceptance criteria — same renderer, header reads "Acceptance criteria". If `acceptanceCriteriaAiSuggested === true` and the user hasn't yet accepted, render a small "AI-suggested" badge next to the heading (the AI Triage slice will toggle the flag to false on accept; out of scope for this design).
- Comments section.

Sidebar (right column):

- Reporter (label + avatar + name + relative time `Created 2h ago`).
- Created / Updated / Closed-at timestamps.
- Labels — read-only chip list.
- A single `Edit` button at the top of the sidebar.
- Delete button at the bottom of the sidebar, only rendered when `me.id === reporter.id || me.id === project.ownerId`. Opens a `<DeleteIssueDialog>` (type the `slug_N` to confirm).

### Edit mode

Same shell, fields become inputs:

| Field                | Control                                                |
|----------------------|--------------------------------------------------------|
| Title                | text input, 1–200                                      |
| Description          | plain `<textarea>` (mono, min-height 160px), 0–10000   |
| Acceptance criteria  | plain `<textarea>` (mono, min-height 96px), 0–10000    |
| Status               | dropdown (5 options)                                   |
| Priority             | dropdown (4 options)                                   |
| Assignee             | `<AssigneePicker>`                                     |
| Labels               | `<LabelPicker>`                                        |

Footer: `Cancel` (ghost) and `Save changes` (accent-1). Save calls `PUT` with the full payload and on success re-emits `IssueFull` upstream — the modal stays open, edit mode exits. Validation errors map back to fields via the shared `errorCode → field` util.

### `<AssigneePicker>`

A combobox: button shows current assignee avatar + name, click opens a popover with a search input. Each keystroke debounced 200ms calls `GET /api/users/search?q=`. Always includes a pinned "Unassigned" row at the top. Selection assigns immediately in edit mode (committed on Save) or calls the assignee PATCH in any future quick-assign UX.

### `<LabelPicker>`

Popover with search input + the project's labels (paged client-side from the labels GET, max 200). Click toggles selection. If no exact match for the typed query, a "+ Create label `name`" row appears — calls POST and selects the result. Each label row has a hover menu (•••) for rename and delete — rename inline (text input swap, Enter commits), delete behind a small confirm. Both gated to project owner client-side and server-side. On rename / delete, the picker re-fetches the labels list to keep counts honest across cards.

### `<CreateIssueModal>` (quick create)

Title-only modal: single text input + `Create` button. Posts to `/api/projects/{slug}/issues`, opens the resulting issue in the detail modal so the user can flow into Edit if they want. Triggered from the viewbar `+ New issue` button.

### Comments section (inside `<IssueDetail>`)

Below the description/criteria:

- Composer pinned at the bottom of the comments area — small `<textarea>` + `Post` button. Empty body disables the button. Cmd/Ctrl+Enter posts, Enter inserts newline (we're markdown).
- Comment list ASC by `createdAt`. Each comment card: avatar + name + relative time + `(edited)` badge + body (rendered markdown). On own comments, a small `•••` reveals `Edit` and `Delete`. Edit swaps the body for a textarea + Save/Cancel; delete uses a small confirm popover.
- "Load more comments" pagination — load first 20 ASC, then a "Load more" button at the bottom of the loaded set to fetch the next page. Comments posted while the user is on the page get appended optimistically on local post.

### Zod schemas (`lib/schemas/issues.ts`)

```ts
export const issueStatus = z.enum(["backlog","todo","in-progress","in-review","done"]);
export const issuePriority = z.enum(["low","medium","high","urgent"]);

export const issueCreateSchema = z.object({
  title: z.string().min(1).max(200),
  description: z.string().max(10000).optional(),
  priority: issuePriority.optional(),
  assigneeId: z.string().optional(),
  labelIds: z.array(z.string()).max(20).optional(),
  acceptanceCriteria: z.string().max(10000).optional(),
});

export const issueUpdateSchema = issueCreateSchema.extend({
  status: issueStatus,
  priority: issuePriority,
});

export const labelSchema = z.object({
  name: z.string().min(1).max(40),
  color: z.string().regex(/^#[0-9A-Fa-f]{6}$/),
});

export const commentSchema = z.object({ body: z.string().min(1).max(10000) });
```

The shared `errorCode → field` mapper gains entries for the new domain codes.

## Validation & error codes

FluentValidation, in-slice:

```csharp
// CreateIssue
RuleFor(x => x.Title).NotEmpty().MaximumLength(200)
    .WithErrorCode("issues:issue:title:required_or_too_long");
RuleFor(x => x.Description).MaximumLength(10_000).When(x => x.Description is not null)
    .WithErrorCode("issues:issue:description:too_long");
RuleFor(x => x.AcceptanceCriteria).MaximumLength(10_000).When(x => x.AcceptanceCriteria is not null)
    .WithErrorCode("issues:issue:acceptance_criteria:too_long");
RuleFor(x => x.Priority).IsInEnum().When(x => x.Priority.HasValue)
    .WithErrorCode("issues:issue:priority:invalid");
RuleFor(x => x.LabelIds).Must(ids => ids is null || ids.Count <= 20)
    .WithErrorCode("issues:issue:labels:too_many");

// UpdateIssue — adds Status (IsInEnum) and reuses the rest.
// ChangeStatus — single Status field, IsInEnum.
// ChangeAssignee — AssigneeId is optional (null clears).
// CreateLabel — Name 1–40, normalized lowercase; Color regex ^#[0-9A-Fa-f]{6}$.
// CreateComment / UpdateComment — Body 1–10_000.
```

Domain-level errors raised from handlers via `DomainException`:

| Code                                                | When                                                   | HTTP |
|-----------------------------------------------------|--------------------------------------------------------|------|
| `issues:issue:not_found`                            | get/edit/delete/status/assign unknown number           | 404  |
| `issues:issue:assignee:not_found`                   | `assigneeId` references missing user                   | 404  |
| `issues:issue:labels:not_in_project`                | any `labelIds[]` entry belongs to a different project  | 400  |
| `issues:issue:delete:forbidden`                     | caller is neither reporter nor project owner           | 403  |
| `issues:issue:status:invalid`                       | enum out of range (defensive; FV catches first)        | 400  |
| `projects:label:name:already_exists`                | unique-violation on label create/rename                | 409  |
| `projects:label:not_found`                          | rename/delete unknown label                            | 404  |
| `projects:label:edit:forbidden`                     | rename/delete by non-owner                             | 403  |
| `comments:comment:not_found`                        | edit/delete unknown comment                            | 404  |
| `comments:comment:update:forbidden`                 | non-author edit                                        | 403  |
| `comments:comment:delete:forbidden`                 | non-author delete                                      | 403  |

All wrapped in RFC 7807 ProblemDetails with `errorCode` + `traceId`. ProblemDetails is passed through the BFF unchanged.

## Out of scope

- AI Triage and AI PR Reviewer (their own designs).
- Real-time updates (board polls on focus; per CLAUDE.md no SignalR).
- Saved filter presets / shareable view URLs beyond `?q=`.
- Bulk operations (multi-select cards, bulk reassign / bulk label / bulk close).
- Sorting controls on the board (fixed `created_at DESC, id DESC`).
- Search across `acceptanceCriteria` (only title + description in `q`).
- Per-issue activity log (status changes, label edits). Comments are the only history surface in MVP.
- Notifications, mentions, watchers.
- Attachments on issues or comments.
- Soft-delete / recovery for issues, labels, or comments.
- Reactions on comments.
- Issue templates.
- "Number → id" backfill for old data — `Number` starts at 1 on the migration since there are no issues yet.
