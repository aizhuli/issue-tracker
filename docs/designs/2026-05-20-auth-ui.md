# Auth UI Design

Date: 2026-05-20
Scope: Next.js frontend auth pages (`/login`, `/register`). No backend changes.

## Overview

The auth pages render a full-viewport animated demo board — blurred and scaled — behind a centered frosted-glass panel. Users see the product working before they log in. The panel contains a tabbed form for Log in / Sign up.

---

## Page Shell & Layout

Both `/login` and `/register` render the same shell component (`AuthShell`). There is no sidebar, topbar, or viewbar — those are post-login chrome.

**Background board:** `DemoBoard` fills the entire viewport (`position: absolute; inset: 0`), rendered with `pointer-events: none`, `user-select: none`, and `filter: blur(5px) saturate(0.85) brightness(0.97)`. `transform: scale(1.06)` fills the edge gaps left by the blur.

**Overlay:** A `rgba(18, 34, 26, 0.18)` tint layer sits between the board and the panel to increase contrast against the frosted glass.

**Auth panel:** Centered via `position: absolute; top: 50%; left: 50%; transform: translate(-50%, -50%)`. Width: `min(520px, 50vw)`, `minWidth: 340px`, `maxHeight: 90vh`, `overflowY: auto`. Frosted glass style: `background: rgba(242, 245, 238, 0.88)`, `backdropFilter: blur(12px) saturate(1.1)`, `border: 1px solid rgba(220, 226, 210, 0.7)`, `borderRadius: var(--radius-lg)`, `boxShadow: var(--shadow-lg)`.

**Brand mark:** A floating pill at `position: absolute; top: 16px; left: 18px`. Renders the app logo SVG + **"Aigo - Issue Board"** in `.sb-ws-name` style. Background: `rgba(255, 255, 255, 0.85)` with `backdropFilter: blur(8px)`. `pointer-events: none`.

**Route protection:** Next.js middleware redirects authenticated users away from `/login` and `/register` immediately.

## Auth Panel Internals

### Header

App logo SVG + "Aigo - Issue Board" in `.sb-ws-name` style at the top of the panel.

### Tab strip

Two tabs using `.dr-tabs` / `.dr-tab` classes:
- **Log in** (`/login`)
- **Sign up** (`/register`)

Active tab: `border-bottom: 2px solid var(--ink-0)`. Switching tabs calls `router.push('/login')` or `router.push('/register')` — no full reload.

### Log in form fields

1. **Email** — labeled input, full width
2. **Password** — labeled input + show/hide toggle (eye icon via `Icon` component), full width
3. **CTA:** "Log in" button — full width, `height: 40px`, `background: var(--accent-1)`, `color: var(--accent-1-ink)`, `border-radius: 9px`

### Sign up form fields

1. **Name** — labeled input, full width
2. **Email** — labeled input, full width
3. **Password** — labeled input + show/hide toggle, full width
4. **CTA:** "Create account" button — same style as Log in CTA

### Inline field errors

Each field shows a small `color: #B94D2F` message below it when validation fails. The field border shifts to `#B94D2F`. Errors are mapped from BFF `errorCode` values (see Error Handling section).

### Form-level errors

Rendered as a pill above the CTA: `background: #F0DDDD`, `color: #7A3535`, `border-radius: 8px`, `padding: 8px 12px`, `font-size: 12.5px`. Used for `auth:credentials:invalid` and network/5xx failures.

### Loading state

CTA disables and shows a CSS `border`-based spinner (no library) while the BFF call is in flight.

### Panel footer

`color: var(--ink-3)`, `font-size: 11px` — "© 2025 Aigo" — pinned to the bottom of the panel.

## Board Animation

Implemented as a single hook `useDemoAnimation(issues, setIssues)`. Two independent `setInterval` timers start on mount and are cleared on unmount. Interval IDs are captured as local constants inside the `useEffect` body and cleared in the cleanup return. `useRef` is used for the new-card count and the pause flag (values that must survive re-renders without triggering them).

### Timer 1 — card move (every 3.5s)

Picks a random issue that is not in `done`, advances it one status step:
```
backlog → todo → in-progress → in-review → done
```
Issues that reach `done` reset to `backlog` after a 2s pause so the board never empties. Applied via `setIssues` — `KanbanBoard` re-renders naturally with its existing CSS transitions.

### Timer 2 — card appear (every 6s)

Inserts a new mock issue (via `createMockIssue`, with `entering: true`) into a random non-`done` column. The `entering` flag triggers the `card-enter` keyframe (`opacity: 0 → 1`, `transform: translateY(-6px) → 0`, 250ms) in `KanbanBoard`. The flag is cleared after 300ms via `setTimeout`. After 4 new cards accumulate the timer pauses for 12s before resuming, preventing board overflow.

## Error Handling & Redirect Flow

### Submission flow

1. Client-side Zod validation runs first. Errors appear instantly without a network call.
2. If Zod passes, the form `POST`s to the BFF route handler (`/api/auth/login` or `/api/auth/register`).
3. The BFF forwards to the ASP.NET API and passes ProblemDetails through unchanged.
4. The frontend reads `errorCode` from the response body and maps it to field-level or form-level messages.

### Error mapping

| `errorCode` | Where shown | Message |
|---|---|---|
| `auth:user:email:required` / `invalid_format` / `too_long` | Below email field | "Enter a valid email address" |
| `auth:user:password:required` / `too_short` / `too_long` | Below password field | "Password must be 8–128 characters" |
| `auth:user:name:required` / `too_long` | Below name field | "Name is required (max 100 chars)" |
| `auth:user:email:already_exists` | Below email field | "An account with this email already exists" |
| `auth:credentials:invalid` | Form-level (above CTA) | "Incorrect email or password" |
| Network / 5xx | Form-level | "Something went wrong — please try again" |

### Redirect flow

- On success the BFF writes the session cookie and returns `{ userId, email, name }`.
- The frontend reads `returnTo` from the query string. If present and starts with `/`, redirects there. Otherwise redirects to `/`.
- Uses `router.push` — no full page reload.

## Zod Schemas (`lib/schemas/auth.ts`)

```ts
export const loginSchema = z.object({
  email: z.string().email().max(254),
  password: z.string().min(1),
});

export const registerSchema = z.object({
  name: z.string().min(1).max(100),
  email: z.string().email().max(254),
  password: z.string().min(8).max(128),
});
```

## File Structure

```
app/
└── (auth)/
    ├── layout.tsx            # passthrough; pages render AuthShell directly
    ├── login/
    │   └── page.tsx          # renders AuthShell with activeTab="login"
    └── register/
        └── page.tsx          # renders AuthShell with activeTab="register"

components/
├── auth/
│   ├── AuthShell.tsx         # full-viewport: blurred DemoBoard + tint overlay + centered frosted-glass AuthPanel
│   ├── DemoBoard.tsx         # KanbanBoard + MOCK_DATA + useDemoAnimation
│   ├── AuthPanel.tsx         # logo, tabs, renders LoginForm or RegisterForm
│   ├── LoginForm.tsx         # email + password fields, submit, error display
│   ├── RegisterForm.tsx      # name + email + password fields, submit, error display
│   └── useDemoAnimation.ts   # two setInterval timers
└── ui/
    ├── FormField.tsx         # label + input + inline error message
    ├── PasswordInput.tsx     # input + show/hide toggle
    └── FormError.tsx         # form-level red pill

lib/
└── schemas/
    └── auth.ts               # loginSchema + registerSchema (Zod)
```

## Out of Scope

- Password reset, email verification, OAuth.
- Remember me / persistent sessions.
- CAPTCHA or rate-limit UI.
- Mobile / responsive layout (demo targets desktop).
- Vitest / Playwright tests for auth UI (deferred to a follow-up step).
