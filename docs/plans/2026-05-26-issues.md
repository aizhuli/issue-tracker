# Implementation Plan: Issues (Kanban board + full CRUD + comments + labels)

Design: `docs/designs/2026-05-26-issues.md`
Depends on: `docs/plans/2026-05-24-projects.md` (Project entity, slug-keyed routing, pagination plumbing, app shell, Modal/Avatar primitives, project list page, owner gating pattern — all in `main`). Also relies on the auth slice (`BffAuth`, `ICurrentUser`, `IdFactory`, `IdEncoding`, `DomainException` family, `ValidationBehavior`, `GlobalExceptionHandler`).

## Context

- Backend feature dirs (`Features/Issues/`, `Features/Comments/`, `Features/Ai/`) exist and are empty. `Features/Projects/` is full; reuse its patterns.
- `Data/Entities/Issue.cs` and `Data/Entities/Label.cs` exist today as **stubs** (`{ Id, ProjectId }`) created in the projects plan so cascade FKs could be wired. They must be replaced — not extended — with the full entities defined in the design. `Project.cs` gains a `NextIssueNumber` column.
- Pagination plumbing (`PageResponse<T>`, `LimitOffsetPaging`, `BadRequestException`) and the test harness conveniences (`AsUser`, `Db`, `CreateUserClient`, `IdFactory` deterministic seed) are already in `main` from the projects plan.
- Frontend already has the app shell (`Sidebar` / `Topbar` / `Viewbar` / `AppShell`), the `Modal` and `Avatar` primitives, the projects list page, and `lib/errors.ts` with `mapProblemDetailsToFields`. The `frontend/components/kanban/KanbanBoard.tsx` prototype and `frontend/lib/mock-issues.ts` are mock-data sketches — **delete them** when the real board lands in Phase 3.
- Backend members follow `vertical-slice-architecture`, `validation`, `error-handling`, `id-generation`, `token-pagination`, and `component-testing`. Frontend members follow `ui-design` and consult `frontend/node_modules/next/dist/docs/` before writing any Next.js 16 code (per `frontend/AGENTS.md`).
- Recommended team size: 4–5 members.

## Phase 2 — Backend slices + Frontend components

*Depends on: Phase 1. Backend slices need 1.1 + 1.5; frontend components need 1.2 + 1.3 + 1.4 + 1.6. 18 parallel tasks. No two tasks share a file.*

Each backend slice follows the `vertical-slice-architecture` skill: one file with nested `Endpoint` / `Body` / `Request` / `Response` / `RequestValidator` / `RequestHandler`. All endpoints under the `AuthPolicies.RequireUser` policy. All IDs in DTOs are Crockford Base32 via `IdEncoding`; raw `long`s never leave the API.

### Task 2.1: `CreateIssue` slice

- [ ] `POST /api/projects/{slug}/issues` with body `{ title, description?, priority?, assigneeId?, labelIds?, acceptanceCriteria? }`. **`status` is never accepted** — always `Backlog`.
- [ ] Validator per design (title 1–200 required; description ≤10000; acceptance criteria ≤10000; priority valid enum when present; labelIds count ≤20).
- [ ] Handler:
  - Resolve project by slug → `NotFoundException("projects:project:not_found")`.
  - Allocate `Number` via raw-SQL update (`UPDATE projects SET next_issue_number = next_issue_number + 1 WHERE id = @id RETURNING next_issue_number - 1`) executed via `Database.SqlQueryRaw<int>` or `ExecuteSqlInterpolatedAsync` returning a value — pick whichever the EF Core 9 docs in `frontend/node_modules` (n/a backend) confirm; for backend confirm with the project's existing EF Core version helpers. Wrap in a transaction with the insert so a failed insert doesn't burn a number? **No** — design accepts burned numbers; just do the update-then-insert without a transaction.
  - Insert the issue with `Id = idFactory.Create()`, `ReporterId = currentUser.UserId`, `Status = Backlog`, `Priority = request.Priority ?? Medium`, `CreatedAt = UpdatedAt = utcNow`.
  - If `labelIds` present: decode each, fetch matching `Label` rows scoped to this project, fail `BadRequestException("issues:issue:labels:not_in_project")` if any decoded id doesn't belong; insert `IssueLabel` rows.
  - If `assigneeId` present: decode + `AnyAsync` on Users; fail `NotFoundException("issues:issue:assignee:not_found")` otherwise.
- [ ] Response: full `IssueFull` DTO (see Task 2.2 for the canonical shape — same record, used by Create/Get/Update/Status/Assignee responses). Set `Location: /api/projects/{slug}/issues/{number}` and return 201.

**Files:** `backend/src/AiIssueTracker.Api/Features/Issues/CreateIssue.cs`

### Task 2.3: `ListIssues` slice (single-column, token-paginated)

- [ ] `GET /api/projects/{slug}/issues?status={s}&pageToken=&maxPageSize=&q=&priority=*&assignee=*` — `status` **required**, single value.
- [ ] Validator: `status` parseable into the enum; `priority` repeatable, each valid; `assignee` repeatable strings (Base32 user id or literal `"unassigned"`).
- [ ] Handler:
  - Resolve project by slug or 404.
  - Build base query filtered by `(ProjectId, Status)`.
  - `q`: case-insensitive substring on `Title` OR `Description` (use `EF.Functions.ILike(x.Title, $"%{q}%")` — yes, surrounding wildcards; per design, not prefix-only).
  - `priority`: OR-in via `.Where(i => priorities.Contains(i.Priority))` when non-empty.
  - `assignee`: split into `unassignedIncluded` + `decodedUserIds`. Build OR predicate: `assigneeIds.Contains(i.AssigneeId.Value) || (unassignedIncluded && i.AssigneeId == null)`.
  - Sort: `OrderByDescending(i => i.CreatedAt).ThenByDescending(i => i.Id)`.
  - Project to `IssueSummary` per design (id, number, displayKey, title, status, priority, assignee summary, labels, commentCount, updatedAt). Compute `commentCount` via subquery: `Comments.Count(c => c.IssueId == i.Id)`.
  - Token round-trip via `LimitOffsetPaging`. Pass `(status, q, priorities, assignees)` as request parameters so token reuse across changed filters is rejected with `paging:validation:page_token_invalid`.

**Files:** `backend/src/AiIssueTracker.Api/Features/Issues/ListIssues.cs`

## Out of scope (deferred per design)

- AI Triage (`POST /api/issues/{id}/ai/triage`) and AI PR Reviewer — separate plans.
- Real-time updates / SignalR / web sockets (board polls only on filter change for MVP).
- Saved filter presets, shareable filter URLs beyond `?q=`.
- Bulk operations (multi-select cards).
- Sort controls (fixed `CreatedAt DESC, Id DESC`).
- Search across `acceptanceCriteria` — `q` covers title + description only.
- Per-issue activity log beyond comments.
- Notifications, mentions, watchers, reactions on comments.
- Attachments on issues or comments.
- Soft-delete / recovery.
- Issue templates.
- Label settings page (managed inline from the picker).
- Number → id backfill — `Number` starts at 1 because there are no existing issues.
