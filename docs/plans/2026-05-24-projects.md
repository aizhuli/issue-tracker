# Implementation Plan: Projects (CRUD with owner-gated edit/delete)

Design: `docs/designs/2026-05-24-projects.md`
Depends on: `docs/plans/2026-05-19-auth.md` (BffAuth, `ICurrentUser`, `IdFactory`, `IdEncoding`, `DomainException` family, `ValidationBehavior`, `GlobalExceptionHandler`, test harness — all already in `main`).

## Context

- Backend feature dirs (`Features/Projects/`) and test dirs (`Tests/Features/Projects/`) already exist but are empty.
- `Common/Pagination/` exists but is empty — the token-pagination plumbing referenced by the design must be built here as foundation.
- Frontend has `(auth)` shell, auth components, and BFF auth routes. The authenticated app shell (sidebar/topbar/viewbar), the modal primitive (`.ai-modal` / `.ai-scrim` / `ai-pop`), and the `Avatar` component referenced by the design **do not exist yet**. They're built here at minimum-viable scope (only what `/projects` needs) — future features can extend.
- Backend team members follow the `vertical-slice-architecture`, `validation`, `error-handling`, `id-generation`, `token-pagination`, and `component-testing` skills. Frontend team members follow `ui-design` and consult `frontend/node_modules/next/dist/docs/` before writing any Next.js 16 code.
- Recommended team size: 4–5 members.

### Task 1.2: Token pagination plumbing + Program.cs registration ✅

- [x] `Common/Pagination/PageResponse.cs` — `public record PageResponse<T>(T[] Items, string? NextPageToken);`
- [x] `Common/Pagination/PagingOptions.cs` — `TokenEncryptionKeyInBase64` (32-byte AES), `TokenIvInBase64` (16-byte), `DefaultMaxPageSize` (10), `MaxMaxPageSize` (100); bind from `Paging` section
- [x] `Common/Pagination/LimitOffsetPaging.cs` — single class with:
    - `TryGetMaxPageSize(int? requested, out int resolved)` — null → default; >max → false; ≤0 → false
    - `TryGetOffsetAndLimit(string? pageToken, int maxPageSize, out int? offset, out int? limit, params object?[] requestParameters)` — null/empty token returns `(0, maxPageSize, true)`; otherwise AES-decrypt + SHA-256 hash-check the request params; returns false if anything is off
    - `CreateNextPageToken(int returnedCount, int offset, int limit, params object?[] requestParameters)` — returns null if `returnedCount < limit`, otherwise AES-encrypts `{offset+limit, sha256(requestParameters)}`
    - Crockford Base32 for the wire format (reuse `IdEncoding` Base32 wrappers from `SimpleBase`)
- [x] `Common/Pagination/PaginationExceptions.cs` — static helpers `InvalidMaxPageSize()` → `BadRequestException("paging:validation:max_page_size_invalid")`, `InvalidPageToken()` → `BadRequestException("paging:validation:page_token_invalid")`. If `BadRequestException` doesn't exist yet, add it to `Common/Exceptions/` alongside the existing `DomainException` family (400 status code).
- [x] In `Program.cs`: `builder.Services.Configure<PagingOptions>(builder.Configuration.GetSection("Paging"));` and `builder.Services.AddSingleton<LimitOffsetPaging>();`
- [x] In `appsettings.Development.json` (or `appsettings.json`) add a `Paging` section with dev-only key + IV (clearly marked) and the default limits

## Phase 2 — Backend slices + Frontend components

*Depends on: Phase 1 (Project entity from 1.1; pagination plumbing from 1.2; UI primitives from 1.5; app shell from 1.6). 10 parallel tasks. No two tasks share a file.*

Each backend slice follows the `vertical-slice-architecture` skill: one file with nested `Endpoint`, optional `Body`, `Request`, `Response`, `RequestValidator`, `RequestHandler`. All endpoints require `AuthPolicies.RequireUser` except where noted.

### Task 2.1: `CreateProject` slice

- [ ] `POST /api/projects` with body `{ name, slug, description? }`; returns 201 with full DTO and `Location: /api/projects/{slug}`
- [ ] `RequestValidator` per design error codes (name 1–100, slug regex, description ≤2000)
- [ ] Handler: lowercase the slug, set `OwnerId = currentUser.UserId`, `CreatedAt = UpdatedAt = TimeProvider.GetUtcNow()`, `Id = idFactory.Create()`, `Add` + `SaveChangesAsync`; on `DbUpdateException` whose inner is a Postgres unique-violation on the slug index throw `ConflictException("projects:project:slug:already_exists")`
- [ ] Response DTO includes `id` (encoded), `slug`, `name`, `description`, `ownerId` (encoded), `ownerName`, `createdAt`, `updatedAt`

**Files:** `backend/src/AiIssueTracker.Api/Features/Projects/CreateProject.cs`

### Task 2.2: `GetProject` slice

- [ ] `GET /api/projects/{slug}` — anyone authenticated can read
- [ ] Handler: lookup by slug with `.Include(p => p.Owner)`, project to full DTO (same shape as create response); throw `NotFoundException("projects:project:not_found")` when missing

**Files:** `backend/src/AiIssueTracker.Api/Features/Projects/GetProject.cs`

### Task 2.3: `ListProjects` slice

- [ ] `GET /api/projects?pageToken=&maxPageSize=&q=` — follows the `token-pagination` skill exactly
- [ ] Filter: `q` (case-insensitive prefix match on name OR slug) — use `EF.Functions.ILike(name, q + '%')` for Postgres
- [ ] Sort: `OrderByDescending(p => p.CreatedAt).ThenByDescending(p => p.Id)` (stable tiebreaker)
- [ ] Response items use `ProjectSummary` DTO: `{ id, slug, name, ownerId, ownerName, createdAt }` — `description` and `updatedAt` are NOT in the summary per design
- [ ] Pass `q` as a request parameter to both `TryGetOffsetAndLimit` and `CreateNextPageToken` so token validation rejects token reuse across different `q` values

**Files:** `backend/src/AiIssueTracker.Api/Features/Projects/ListProjects.cs`

### Task 2.4: `UpdateProject` slice

- [ ] `PUT /api/projects/{slug}` with body `{ name, description }` (no slug field — slug is identifier only)
- [ ] Validator: same name/description rules as create
- [ ] Handler: lookup by slug; if missing throw `NotFoundException("projects:project:not_found")`; if `project.OwnerId != currentUser.UserId` throw `ForbiddenException("projects:project:edit:forbidden")` (add `ForbiddenException` to `Common/Exceptions/` if it doesn't exist — 403); update `Name`, `Description`, bump `UpdatedAt = TimeProvider.GetUtcNow()`; never touch `Slug` or `OwnerId`; save and return the same full DTO shape as `GetProject`

**Files:** `backend/src/AiIssueTracker.Api/Features/Projects/UpdateProject.cs`

### Task 2.5: `DeleteProject` slice

- [ ] `DELETE /api/projects/{slug}` — returns 204
- [ ] Handler: lookup by slug; 404 if missing; 403 with `projects:project:delete:forbidden` if not owner; `Remove` + `SaveChangesAsync` (cascade FKs configured in 1.1 take care of children)

**Files:** `backend/src/AiIssueTracker.Api/Features/Projects/DeleteProject.cs`

### Task 2.6: `SlugAvailability` slice

- [ ] `GET /api/projects/slug-availability?slug={s}` — always 200, never 400
- [ ] Validator: only that `slug` query is non-empty (`ArgumentNullException`/400 if entirely missing is acceptable — required query string)
- [ ] Handler: if slug fails the regex from the design, return `{ slug, available: false, reason: "invalid_format" }`; else `AnyAsync` on the slug column; if taken return `{ slug, available: false, reason: "taken" }`; else `{ slug, available: true }`
- [ ] No write, no auth check beyond `RequireUser`

**Files:** `backend/src/AiIssueTracker.Api/Features/Projects/SlugAvailability.cs`

### Task 2.7: Frontend `<ProjectCard>` component

- [ ] `frontend/components/projects/ProjectCard.tsx` — pure presentational
- [ ] Props: `{ project: ProjectSummary, me: { id: string } | null, onEdit?: () => void, onDelete?: () => void }`
- [ ] Renders the card per the design (Manrope name, mono slug chip, 3-line clamped description preview — note: summary has no description, so the card uses a placeholder or omits description; if omitted update the design later)
- [ ] Bottom-right `dots` icon menu (use existing `Icon name="dots"` — add to the icon set if missing) only when `project.ownerId === me?.id`; menu items "Edit" and "Delete" call the respective callbacks

**Files:** `frontend/components/projects/ProjectCard.tsx`

### Task 2.8: Frontend `<ProjectFormModal>` component

- [ ] `frontend/components/projects/ProjectFormModal.tsx` — wraps `Modal` from 1.5
- [ ] Props: `{ open: boolean, mode: "create" | "edit", project?: ProjectFull, onClose: () => void, onSaved: (project) => void }`
- [ ] State: `name`, `slug`, `description`, per-field errors, `slugStatus: "idle" | "checking" | "available" | "taken" | "invalid_format"`, `submitting`
- [ ] Slug auto-fill: a `slugTouched` ref starts false; `onChange` of name updates slug via a `slugify(name)` helper until `slugTouched` flips to true (set when user types in slug input)
- [ ] Slug availability: on every slug change, run local Zod; if invalid set `invalid_format` and skip the BE call; if valid debounce 400ms then call `GET /api/projects/slug-availability` via an `AbortController` stored in a `useRef` and aborted on next keystroke; map response → status
- [ ] UI states next to slug input: muted "Checking…" with spinner / red format hint / green check + "Available" / red "Already taken — pick a different one". In `edit` mode slug input is read-only and shows "(cannot be changed)" — no availability check.
- [ ] Submit button disabled while `submitting`, while `slugStatus !== "available"` (create mode), or when local Zod fails
- [ ] Submit: `POST /api/projects` (create) or `PUT /api/projects/{slug}` (edit) with the JSON body. On 4xx parse `errorCode` and route to the relevant field via the helper from Task 1.4; specifically `projects:project:slug:already_exists` flips `slugStatus = "taken"` without any special-case branch. On 2xx call `onSaved(body)` then `onClose()`.

**Files:** `frontend/components/projects/ProjectFormModal.tsx`

- Project transfer / change-owner
- Soft delete or recoverable trash
- Project membership / per-user roles / non-owner edit/delete
- Slug rename + redirect from the old slug
- Rate-limiting the slug-availability endpoint
- Server-side full-text search (`?q=` stays prefix-only for MVP)
- E2E Playwright flow (the manual smoke pass in 3.9 is enough for MVP)
- Migrating the auth forms to the shared `mapProblemDetailsToFields` helper if Task 1.4 had to create it
- Fleshing out the `Issue` and `Label` entities beyond the FK stubs (those are owned by the next vertical slices)
